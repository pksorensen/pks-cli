# PKS CLI — Draft Feature List (Step 1 + revision pass)

Generated by static code walk of `src/Commands/`, `src/Infrastructure/Services/`, `src/Program.cs`, and `tests/`. Revised after human review with answers to the open questions.

## Method

Walked the command tree registered in `src/Program.cs` (45 top-level command folders under `src/Commands/`) and grouped by **shared infrastructure + product surface** rather than per-CLI-verb. Granularity target was 10–15 FTs.

**Revision pass** (human input):
- **Foundry** split into two FTs: auth/resource-selection vs. the proxy. The proxy is a security-boundary capability so devcontainers / claude / spawn can use Foundry without exposing the user's full credentials — load-bearing for the agentic substrate.
- **MCP and Exec** split. Exec is its own protocol for tools requesting AI access, not just an MCP variant.
- **Agent** dropped from FT-Agentics. The old `Agent Framework` folder is deprecated; only `Agentics` (the runner/tasks model for the agentics platform) is first-class.
- **Brain** description elevated: it's a cross-project capability for end users (ingest sessions, derive features/ADRs/wiki, search). Not just self-documentation of pks-cli.
- **Vibecast** reframed: it's a separate tool **tightly integrated** with pks-cli because it's the **operator of the ALP spec** within the agentic runner. The clustering with Voice stays for now (both surface broadcasts), but the operator role is the primary framing.

Result: **17 FTs**, slightly above the original target band but architecturally justified by the two splits.

## Domains observed

- **agent-infra** — devcontainers, VMs, firecracker microVMs, SSH targets, rsync, fileshare
- **agentic-runtime** — agentics runner/tasks, MCP servers, exec protocol
- **ai-providers** — Azure AI Foundry (+ proxy), Azure billing, Google AI, Claude usage/stats, model registry, TTS/image gen
- **knowledge** — Brain (cross-project), PRD, promptwall, skills
- **broadcast / alp-operator** — Vibecast (ALP spec operator), Voice (heypoul)
- **integrations-ticketing** — ADO, Jira, Confluence, GitHub
- **integrations-identity-data** — MS Graph, Google, Email export
- **observability** — AppInsights, OTEL, structured logging, telemetry
- **claude-hooks** — hooks dispatcher, managed settings
- **plugin-marketplace** — marketplace add/list/show/enable/disable/refresh
- **cli-core** — `init`, `deploy`, `status`, `ascii`, `tools publish`, project identity

## Resolved open questions

1. **Brain motivation** — RESOLVED. Generic feature for all projects of pks-cli's users; this repo is just the first consumer. Description updated.
2. **Vibecast first-class?** — RESOLVED. First-class, separate tool but tightly integrated as the operator of the ALP spec inside the agentic runner. Stays as its own FT.
3. **Agentics vs Agent** — RESOLVED. Drop Agent (deprecated). Agentics is the runner concept for the agentics platform. Kept as one FT (FT-8).
4. **Foundry proxy separate?** — RESOLVED. Yes — split. The proxy is a security-boundary capability for devcontainers/claude/spawn to access Foundry without full credential exposure. Now FT-5.
5. **Hooks vs MCP vs Exec** — RESOLVED. Three separate FTs. Hooks is Claude lifecycle dispatch (FT-11). MCP is tool hosting (FT-9). Exec is a protocol for tools to request AI access (FT-10).
